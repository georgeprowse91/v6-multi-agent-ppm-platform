# Disaster Recovery Runbook

## Overview
This runbook covers automated database backups and restore procedures for the PPM platform.
Nightly backups are generated by the `db-backup` CronJob and uploaded to Azure Blob Storage.

## Backup Automation
- **Schedule:** Daily at 02:00 UTC via `infra/kubernetes/db-backup-cronjob.yaml`.
- **Execution:** The CronJob runs `/scripts/db_backup.sh` from the `db-backup-scripts` ConfigMap.
- **Storage:** Backups are uploaded to the configured Azure Blob Storage container.

### Required Configuration
The `db-backup-config` Secret must include:
- `DB_URL`: PostgreSQL connection string used by `pg_dump`.
- `STORAGE_ACCOUNT`: Azure Storage account name.
- `CONTAINER`: Azure Blob container for backups.
- `AZURE_STORAGE_KEY`: Storage account key or equivalent credentials.

## Restore Procedure
1. **Identify the backup**
   - Locate the desired backup in the Azure Blob Storage container.
   - Example naming format: `db-YYYYMMDDHHMMSS.sql.gz`.

2. **Download the backup**
   ```bash
   az storage blob download \
     --account-name "$STORAGE_ACCOUNT" \
     --container-name "$CONTAINER" \
     --name "db-YYYYMMDDHHMMSS.sql.gz" \
     --file /tmp/db-restore.sql.gz
   ```

3. **Restore to PostgreSQL**
   ```bash
   gunzip -c /tmp/db-restore.sql.gz | psql "$DB_URL"
   ```

4. **Validate the restore**
   - Confirm application health checks are passing.
   - Verify key tables and recent records in the database.

## Operational Notes
- Ensure the CronJob image includes `pg_dump` and `gzip` dependencies. If not, use a custom image.
- Restrict access to the backup container with least-privilege credentials.
- Rotate backup credentials and storage keys per security policy.
