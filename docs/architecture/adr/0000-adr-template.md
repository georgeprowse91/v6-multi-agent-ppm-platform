## ADR Usage Guidelines
### When to Create an ADR
Create an ADR for decisions that:
1. Affect multiple components or teams.
2. Influence system quality attributes (performance, security, scalability).
3. Establish standards or patterns.
4. Resolve major trade-offs or reverse prior decisions.

### Workflow
1. Draft (status: Proposed)
2. Review (stakeholder feedback)
3. Decision (status updated and approvals logged)
4. Implementation (reference ADR in work items)
5. Maintenance (mark deprecated/superseded when needed)

### Organization & Storage
- Use sequential numbering (ADR-0001, ADR-0002, etc.).
- Store in `docs/architecture/adr/` and link from related docs.
- Reference related ADRs and requirements explicitly.
